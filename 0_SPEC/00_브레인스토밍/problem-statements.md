# Problem Statements
## 업무 개선 보드 시스템

---

## 1. 배경 (Background)

### 1.1 현재 상황
- 회사 내 각 부서에서 업무 개선 아이디어와 과제가 발생하고 있음
- 개선 업무들이 체계적으로 관리되지 않고 산발적으로 처리됨
- 경영진이 전사적인 개선 활동 현황을 파악하기 어려움
- 화이트보드나 포스트잇을 활용한 오프라인 방식의 한계 존재

### 1.2 이해관계자
| 이해관계자 | 역할 | 관심사 |
|------------|------|--------|
| 경영자/임원 | 의사결정자 | 전사 개선 활동 흐름 파악, 방치된 과제 식별 |
| 부서 담당자 | 관리자 | 부서 내 개선 과제 관리 |
| 일반 직원 | 등록자 | 개선 아이디어 제안, 불편사항 공유 |

---

## 2. 문제 정의 (Problem Definition)

### 2.1 핵심 문제

> **"회사 내 업무 개선 아이디어와 과제들이 어디서 어떤 상태로 진행되고 있는지 한눈에 파악할 수 없다."**

### 2.2 세부 문제

| # | 문제 | 영향 |
|---|------|------|
| P1 | 개선 아이디어가 등록될 공식적인 창구가 없음 | 좋은 아이디어가 묻히거나 잊혀짐 |
| P2 | 부서별 개선 활동 현황이 파편화되어 있음 | 경영진의 전사 현황 파악 불가 |
| P3 | 개선 과제의 진행 상태를 추적하기 어려움 | 방치되는 과제 발생 |
| P4 | 어떤 부서에서 어떤 고민을 하는지 공유 안됨 | 부서 간 협업 기회 상실 |
| P5 | 완료된 개선 활동이 기록으로 남지 않음 | 성과 및 노하우 축적 불가 |

### 2.3 문제의 근본 원인

```
┌─────────────────────────────────────────────────────────────┐
│                      근본 원인 분석                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   [등록 창구 부재]  →  [개선 아이디어 산실]                  │
│          ↓                                                  │
│   [통합 관리 시스템 없음]  →  [현황 파악 불가]               │
│          ↓                                                  │
│   [진행 상태 추적 어려움]  →  [과제 방치]                    │
│          ↓                                                  │
│   [경영진 의사결정 지연]                                     │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 3. 경영자 관점의 핵심 질문 (Key Questions)

경영자가 답을 얻고 싶어하는 5가지 질문:

| # | 질문 | 현재 상태 |
|---|------|-----------|
| Q1 | "요즘 어디 부서에서 어떤 얘기들이 많이 올라오지?" | ❌ 파악 불가 |
| Q2 | "이 중에서 아직 손도 못 댄 건 뭐야?" | ❌ 확인 불가 |
| Q3 | "오래 걸리는 건 왜 오래 걸릴까?" | ❌ 추적 불가 |
| Q4 | "이건 굳이 지금 안 해도 되는 거 아냐?" | ❌ 판단 근거 없음 |
| Q5 | "이미 끝난 건 어떤 식으로 정리됐지?" | ❌ 기록 없음 |

---

## 4. 제약 조건 (Constraints)

### 4.1 반드시 피해야 할 것 (Anti-Requirements)

| # | 제약 | 이유 |
|---|------|------|
| C1 | ❌ KPI, 점수, 평가 기능 | 성과 평가 목적이 아님 |
| C2 | ❌ 부서 간 성과 랭킹 | 비교/경쟁 유발 방지 |
| C3 | ❌ 개인 평가 연동 | 자유로운 아이디어 제안 저해 |
| C4 | ❌ 달성률/퍼센트 표시 | 정량적 압박감 배제 |
| C5 | ❌ 딱딱한 업무 용어 | 캐주얼한 분위기 유지 |

### 4.2 핵심 원칙

> 📌 **"성과관리가 아닌 리스트 관리"**
> 
> 📌 **"숫자는 '정보'지 '평가'가 아님"**
> 
> 📌 **"화이트보드 + 포스트잇을 온라인으로 옮긴 느낌"**

---

## 5. 목표 상태 (Target State)

### 5.1 비전

> **"현장의 체온을 느낄 수 있는 캐주얼한 업무 개선 보드"**

### 5.2 목표 달성 시 기대 효과

| # | 기대 효과 | 설명 |
|---|----------|------|
| G1 | 현장 체온 감지 | 보고 없이도 현장의 고민과 개선 활동 파악 |
| G2 | 흐름 파악 | 어느 부서에서 무엇을 고민 중인지 한눈에 확인 |
| G3 | 근거 확보 | "이거 누가 한번 봐줘" 한마디의 데이터 근거 |
| G4 | 조직 문화 메시지 | "문제를 없애는 조직"이라는 메시지 전달 |
| G5 | 방치 방지 | 오래된 과제, 멈춤 상태 과제 가시화 |

### 5.3 목표 상태 체크리스트

- [ ] 직원이 부서별로 개선거리(메모 수준)를 자유롭게 등록할 수 있다
- [ ] 개선 항목이 카드 형태로 리스트/보드에 표시된다
- [ ] 각 항목은 간단한 상태(아이디어/보는중/진행중/보류/완료)를 가진다
- [ ] 경영자가 전체 또는 부서별로 항목을 훑어볼 수 있다
- [ ] 성과 지표나 평가 목적의 기능은 포함하지 않는다

## 6. 해결 방안 (Proposed Solution)

### 6.1 솔루션 개요

**"업무 개선 보드" 웹서비스 구축**

```
┌─────────────────────────────────────────────────────────────┐
│                       업무 개선 보드                         │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   [등록]           [조회/필터]           [대시보드]          │
│   - 제목           - 부서별 탭           - 전체 현황         │
│   - 메모           - 상태별 필터         - 부서별 개수       │
│   - 부서(자동)     - 최신순 정렬         - 오래된 항목       │
│                                                             │
│   💡→👀→🛠️→⏸️→✅                                          │
│   떠올림→보고있음→만지는중→멈춤→정리됨                       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 6.2 핵심 기능

| 기능 | 설명 | 우선순위 |
|------|------|----------|
| 개선 항목 등록 | 제목, 메모, 부서(자동) 입력으로 간편 등록 | 필수 |
| 상태 관리 | 💡👀🛠️⏸️✅ 아이콘 기반 캐주얼 상태 | 필수 |
| 부서별 보드 | 탭 형태로 부서별 카드 리스트 조회 | 필수 |
| 대시보드 | 전체/정리됨/멈춤 건수, 부서별 현황 | 필수 |
| 오래된 항목 표시 | 30일+ 변경 없는 항목 하이라이트 | 권장 |
| 파일 첨부 | 참고 사진/문서 업로드 | 선택 |

### 6.3 UX 원칙

| 원칙 | 적용 |
|------|------|
| "나중에 다듬자" | 최소 필수 입력, 등록 버튼: [일단 올리기] |
| "왜 안 적었냐" 방지 | 선택 항목은 진짜 선택, 빈칸 허용 |
| 훑어보는 느낌 | 분석 차트 ❌ → 구경하는 지도 ✅ |
| 캐주얼 톤앤매너 | 관리/성과/평가 용어 회피 |

## 7. 성공 기준 (Success Criteria)

### 7.1 정성적 기준

| # | 기준 | 측정 방법 |
|---|------|-----------|
| S1 | 경영진이 전사 개선 현황을 한눈에 파악할 수 있다 | 경영진 피드백 |
| S2 | 직원들이 부담 없이 아이디어를 등록한다 | 등록 빈도 관찰 |
| S3 | 방치된 과제가 가시화된다 | 오래된 항목 리스트 확인 |
| S4 | 시스템 사용에 별도 교육이 필요 없다 | 사용자 반응 |

### 7.2 정량적 기준 (참고용, 평가 목적 아님)

| # | 지표 | 목표 | 비고 |
|---|------|------|------|
| M1 | 월간 등록 건수 | - | 숫자 자체가 목표 아님 |
| M2 | 활성 사용 부서 수 | 전체 부서 | 모든 부서 참여 |
| M3 | 평균 응답 시간 | 3초 이내 | 시스템 성능 |

## 8. 범위 (Scope)

### 8.1 In-Scope (포함)

- ✅ 개선 항목 등록/조회/수정/삭제
- ✅ 캐주얼 상태 관리 (5단계)
- ✅ 부서별 분류 및 필터링
- ✅ 경영자용 대시보드
- ✅ 기본 사용자 인증

### 8.2 Out-of-Scope (제외)

- ❌ KPI/점수/평가 기능
- ❌ 부서 간 성과 비교/랭킹
- ❌ 개인 평가 연동
- ❌ 복잡한 워크플로우/승인 프로세스
- ❌ 알림/리마인더 (Phase 1)

## 9. 용어 정의 (Glossary)

| 용어 | 정의 |
|------|------|
| 개선 항목 | 직원이 등록하는 업무 개선 아이디어/불편사항/과제 |
| 카드 | 개선 항목을 포스트잇처럼 시각화한 UI 요소 |
| 보드 | 카드들이 모여있는 화면 (화이트보드 느낌) |
| 상태 | 개선 항목의 진행 단계 (💡👀🛠️⏸️✅) |
| 훑어보기 | 분석이 아닌 가볍게 현황을 파악하는 행위 |
