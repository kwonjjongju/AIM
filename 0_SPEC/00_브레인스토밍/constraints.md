# 📋 제약사항 (Constraints)
## 업무 개선 보드 시스템

---

## 1. 비즈니스 제약사항

### 1.1 반드시 피해야 할 것 (Anti-Requirements)

> ⚠️ **이 프로젝트의 핵심 철학: "성과관리가 아닌 리스트 관리"**

| # | 제약 | 상세 설명 | 이유 |
|---|------|----------|------|
| BC-01 | ❌ KPI/점수/평가 기능 | 성과 측정을 위한 어떤 지표도 포함하지 않음 | 성과 평가 목적이 아님 |
| BC-02 | ❌ 부서 간 성과 랭킹 | 부서별 비교/순위 기능 금지 | 경쟁 유발 방지 |
| BC-03 | ❌ 개인 평가 연동 | 인사 시스템과의 어떤 연계도 금지 | 자유로운 제안 저해 |
| BC-04 | ❌ 달성률/퍼센트 표시 | 정량적 진척도 표현 금지 | 압박감 배제 |
| BC-05 | ❌ 복잡한 승인 프로세스 | 다단계 결재/승인 워크플로우 금지 | 간편한 등록 지향 |

### 1.2 톤앤매너 제약

| # | 제약 | 대안 |
|---|------|------|
| TM-01 | '관리' 단어 사용 금지 | → '보드', '리스트' |
| TM-02 | '성과' 단어 사용 금지 | → '정리됨', '완료' |
| TM-03 | '평가' 단어 사용 금지 | → 사용 안함 |
| TM-04 | '분석' 단어 사용 금지 | → '훑어보기', '현황' |
| TM-05 | 딱딱한 상태 용어 금지 | → 캐주얼 상태 (💡👀🛠️⏸️✅) |

### 1.3 사용자 경험 제약

| # | 제약 | 상세 |
|---|------|------|
| UX-01 | 필수 입력 최소화 | 제목, 부서(자동)만 필수 |
| UX-02 | 빈칸 허용 | "왜 안 적었냐" 질문 방지 |
| UX-03 | 별도 교육 불필요 | 직관적 사용 가능해야 함 |
| UX-04 | 등록 버튼 문구 | [일단 올리기] - 부담 없는 느낌 |
